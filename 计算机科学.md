### TCP 与 HTTP 的关系

一句话总结：TCP 是送货卡车负责可靠送达，HTTP 是装箱单描述货品内容——分层各管各的事。

- **层级不同**：HTTP 是应用层协议（OSI 第 7 层），TCP 是传输层协议（OSI 第 4 层）。HTTP 建立在 TCP 之上
- **封装关系**：HTTP 报文被封装在 TCP 数据段中，TCP 段再被封装进 IP 包，逐层向下包装
- **TCP 为 HTTP 提供可靠传输**：保证数据不丢、不重、不乱序，还提供流量控制和拥塞控制
- **三次握手**：HTTP 请求发出前，TCP 已先完成三次握手，建立连接
- **不是绑定关系**：HTTP/1.0 ~ HTTP/2 都依赖 TCP，但 HTTP/3 改用基于 UDP 的 QUIC 协议。HTTP 只需要底层提供可靠传输，TCP 只是历史上最合适的选择
- **形象类比**：TCP 是送货卡车（负责可靠送达），HTTP 是装箱单（描述货品内容和操作指令）

### 纯 TCP 传数据的局限 —— 为什么需要 HTTP

一句话总结：TCP 只传无结构字节流，HTTP 定义了全世界通用的对话语义——自己用 TCP 造轮子不如用现成的标准。

- **TCP 只传输无结构字节流**，对方收到后不知道字节的含义（是什么资源、成功还是失败、什么数据类型）
- **缺少请求语义**：没有 URL、Method（GET/POST）、状态码等概念，每个应用都要自己定义一套——路径格式、成功/失败编码、数据长度分隔方式
- **缺少元数据携带能力**：没有 Header 机制，无法传递 Content-Type、Content-Length、Cache-Control、Authorization 等信息，每次都要自己约定
- **无法互通**：如果每个公司和应用都自定义协议格式，A 的客户端无法与 B 的服务端通信——HTTP 提供了全世界统一的"对话语言"
- **核心原理——分层职责不同**：TCP 解决"怎么可靠送达"（传输层），HTTP 解决"对话的语义是什么"（应用层）。电话线 vs 电话礼仪的类比
- **HTTP/3 的佐证**：HTTP/3 抛弃 TCP 改用 QUIC（UDP），说明 HTTP 的本质是一套语义规范（方法、状态码、Header），不依赖具体传输层——只要底层能可靠传数据就行

### 车载场景：纯 TCP 传结构体的隐患

一句话总结：二进制结构体跨端直传看似高效，实则埋了内存对齐、字节序、版本演进三颗雷，HTTP+JSON 可读可扩展更省心。

- **内存对齐陷阱**：C/C++ 结构体中 `bool` 后编译器可能插入 3～7 字节填充，`sizeof` 结果与直觉不符；Android ARM 与座舱 SoC 的 `sizeof(bool)` 可能不同（1 vs 4）
- **字节序风险**：跨架构（如 ARM ↔ x86）直接传二进制结构体时，多字节字段的字节序可能相反
- **版本演进灾难**：结构体增加字段后，新旧端混用会导致字段错位、后续字段全部误读——二进制格式没有"版本号"，接收端无法判断数据是旧格式还是新格式
- **调试困难**：抓包看到十六进制 dump，无法肉眼判断正误；相比之下 HTTP + JSON 直接人类可读
- **HTTP 可作为替代**：座舱域起 HTTP Server，Android 发 POST 请求带 JSON body；状态码 + Header 机制解决所有上述问题。车内局域网带宽充足，HTTP 额外开销不构成瓶颈
- **其他车载通信方案**：gRPC（HTTP/2 + protobuf，车载 SOA 常用）、MQTT（发布-订阅/传感器流）、SOME/IP（AUTOSAR 体系的服务发现和 RPC）

### gRPC 详解与 DDS 对比

一句话总结：gRPC 是"打电话找特定的人办事"（远程函数调用），DDS 是"广播里喊话谁需要谁听"（全局数据空间）——前者适合明确请求-响应对，后者适合持续信号流。

#### gRPC 是什么

- **远程过程调用（RPC）框架**：让你调用远程服务像调用本地函数一样，Google 开源
- **三大部件**：Protocol Buffers（IDL + 二进制序列化，比 JSON 小 3-10 倍）+ HTTP/2（多路复用、头部压缩、双向流）+ 代码生成器（`.proto` → 10+ 种语言的 Client/Server 代码）
- **四种模式**：Unary（一问一答）、Server Streaming（一问多答）、Client Streaming（多问一答）、Bidirectional（双向流）
- **强类型 `.proto` 契约**：定义 `service` 和 `rpc`，两端自动生成代码，彻底解决结构体手动对齐问题

#### gRPC vs DDS 核心差异

- **通信范式**：gRPC 是 Client→Server RPC（知道地址、调方法签名），DDS 是 Publisher⇢Subscriber（彼此透明，靠 Topic 连接）
- **发现机制**：gRPC 需要外部服务发现（DNS/etcd/Consul），DDS 内置动态发现——节点上线自动感知
- **QoS**：gRPC 无（依赖 HTTP/2 + TCP 尽力而为），DDS 有丰富的 Deadline / Reliability / Durability / Liveness 保障
- **传输层**：gRPC = HTTP/2 over TCP，DDS = DDSI-RTPS over UDP（多播+单播）
- **零拷贝**：gRPC 不支持，DDS 支持同 SoC 内共享内存，数据不动只传指针
- **耦合度**：gRPC 紧耦合——Server 挂了服务停，DDS 松耦合——任何节点可随时上下线

#### 选型规则

- App 发控制指令 → gRPC（天然请求-响应 + 强类型保证）
- 传感器数据洪流 → DDS（一对多广播 O(1)，N 个订阅者不会变为 N 次 RPC）
- 节点随时可能上下线 → DDS（内置动态发现，无需额外配置）
- 跨语言跨团队协作、移动端开发 → gRPC（10+ 语言生态 + Android 原生支持）
