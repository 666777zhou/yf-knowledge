## EDR 系统

</summary>

### YfMiner：自动驾驶 EDR 触发与数据闭环

用户项目：https://github.com/666777zhou/YfMiner

#### 核心思路

- 自动驾驶数据闭环：采集 real data → 检测高价值场景 → 触发 EDR 录制 → 回传数据 → 训练模型 → 更好地检测 → 更精准的触发
- 目标是解决长尾问题：大部分驾驶里程是"平淡"的，真正的急刹、急转弯等高风险场景是小概率事件，需要从海量数据中精准挖掘

#### 级联架构（Cascade）

- **Stage 1 — IMU 异常检测器（136K 参数）**：CNN+LSTM，只吃 IMU 信号（2 秒窗口），功耗毫瓦级，全时运行。召回率 ≥ 95%，宁可多报不可漏报
- **Stage 2 — 多模态融合确认（1.18M 参数）**：MobileNetV3 + PointNet + IMU，融合前视图像 + LiDAR + IMU。精度 ≥ 90%，仅在 Stage 1 报警时才唤醒，负责过滤误报
- 关键设计：轻量级常驻检测 + 高精度按需确认 = 低功耗 + 高准确

#### 伪标注（Pseudo-Labeling）

- KITTI 是正常城市驾驶数据，没有真实碰撞标签，所以用物理阈值规则打伪标签
- 急刹车：前向加速度 < -1.5 m/s² 持续 2 帧
- 急加速：前向加速度 > 2.0 m/s² 持续 2 帧
- 急转弯：横向加速度 > 2.0 m/s² 或横摆角速度 > 0.3 rad/s
- 阈值在 `config.py` 中可调，适应不同驾驶场景（高速、越野等）

#### 部署工具链

- PyTorch 训练 → ONNX 导出 → 高通 QNN SDK 编译 → INT8 量化 → `.dlc` 文件 → SA8650 NPU 运行
- INT8 量化：将 32 位浮点数参数压缩到 8 位整数，推理速度提升 2-4 倍，功耗大幅下降，在 NPU 上运行
- 端侧推理的强约束：模型体积小（< 2MB），功耗低（毫瓦级常驻），实时性要求高

#### 数据源

- KITTI RAW：OXTS GPS/IMU（30 维遥测，10Hz）、四路摄像头、Velodyne 64 线 LiDAR
- 未来方向：接入更多数据源（用户上传的驾驶数据），持续扩充高价值场景库

## 仿真数据源

</summary>

### Omni-Dreams 与 Cosmos

长尾场景在真实驾驶中天然稀少，仿真可以生成这些稀缺数据。

#### NVIDIA Cosmos（2025-2026）

世界基础模型（World Foundation Model）家族，为物理 AI 生成逼真的合成数据：

- **Cosmos Transfer**：将结构化输入（语义分割图、深度图、LiDAR、HD Map、3D 框）转换为逼真视频，可控天气/光照/时段
- **Cosmos Predict 2.5**：预测未来世界状态（最长 30 秒），微调后精度可提升 10 倍
- **Cosmos Reason 2**：时空推理 + Chain-of-Thought，256K token 上下文窗口
- **Cosmos 3（2026.5.31）**：统一视觉、推理、动作生成的"全模态模型（omnimodel）"
- **NuRec / InstantNuRec**：将真实车队采集的数据（摄像头 + LiDAR）重建为 3D 高斯可编辑场景，无需逐场景优化
- **Cosmos-Drive-Dreams**（2025.6 开源）：生成 81,802 条合成驾驶视频，附带 HD Map 和 LiDAR，覆盖雨/雪/雾/夜间
- 已有落地：理想汽车、千哩科技、元戎启行每天生成 1000+ 重建场景和 30 万+ 渲染/仿真

#### NVIDIA OmniDreams（2026.6.2 发布）

实时生成式世界模型，专为自动驾驶闭环仿真设计：

- **核心能力**：根据驾驶动作（action）实时生成下一帧多摄像头画面——方向盘打多少、画面就变多少
- **闭环交互**：AV 策略模型的决策 → 更新仿真状态 → 影响下一轮传感器观测 → 模型再决策……形成完整反馈回路
- **关键参数**：2B 参数，基于 Cosmos-Predict2.5（Diffusion Transformer），训练数据 21000 小时驾驶场景，推理 ~52 FPS @ 720p（1×GB300）
- **三级训练流程**：双向 HD Map 条件教师模型 → 因果 Diffusion Forcing 学生模型 → Self-Forcing + DMD 蒸馏加速
- **长尾场景**：可合成极端天气、异常智能体行为等传统仿真器或真实数据覆盖不了的罕见边缘场景
- **WAM 变体**：从 OmniDreams 后训练的 World-Action Model 在 NuRec 数据集上超越 VLA 模型 Alpamayo 1.5，参数量仅为其 1/5
- **开源**：GitHub (nv-tlabs/omni-dreams) + HuggingFace，NVIDIA Open Model License

#### 为数据闭环提供的价值

- 解决长尾数据稀缺：万分之一概率的急刹/侧翻/鬼探头，仿真可以无限生成
- 补充 EDR 触发前的场景多样性：KITTI 只有城市正常驾驶，仿真可扩展极端场景
- 策略闭环测试：OmniDreams 不是"放视频"，而是让模型真正"开"在仿真世界里，发现模型的盲区
- YfMiner 的潜在结合点：用 Cosmos 生成稀有长尾场景 → 输入 YfMiner Stage1+Stage2 测试触发率 → 发现漏报 → 优化模型 → 再次测试

#### 底层技术概念

- **世界模型（World Model）**：学习世界如何运转的模型——给它当前状态+动作，它预测下一刻的世界状态。对比：传统视频生成只管"好不好看"，世界模型要求"对不对"——刹车灯亮了吗？前车减速了吗？
- **Diffusion Transformer（DiT）**：扩散模型 + Transformer 的结合。扩散过程：给图像逐步加噪→模型学习反向去噪→从纯噪声生成新图像；Transformer：用自注意力机制建模全局关系，替代传统 U-Net
- **Diffusion Forcing**：让扩散模型按时间顺序自回归生成，每一帧的生成以过去帧和当前动作为条件

## 车端通信

</summary>

### DDS vs HTTP

一句话总结：DDS 的严格对齐不是缺陷是性能基石，HTTP 适合低频指令，DDS 适合传感器洪流——车里两者都要用，不是替代关系。

#### 核心差异：通信模式不同

- **HTTP 是请求-响应模式**（一问一答），**DDS 是发布-订阅模式**（持续广播），车辆信号（如车速、GPS）是持续产生的传感器流，不是"有人请求才发送"
- 车速信号一秒发 100 次，10 个消费者用 HTTP 轮询就是 1000 次请求/秒——全是重复拉取；DDS 一个 Topic 广播一次，订阅者各收各的
- 激光雷达点云每秒数百万个点，HTTP + JSON 序列化能把整个 SoC 吃满；DDS 二进制直达内存

#### "严格对齐"是刻意设计，不是缺陷

- **IDL 定义 → 代码生成 → 零拷贝内存映射**：你写的是字段名（`float speed_kmh`），代码生成器自动处理对齐，不是你手填字节偏移
- **8 字节车速 vs 50 字节 JSON**：DDS 没有字符串解析开销、没有 GC 压力、没有 JSON 序列化
- **确定性延迟**：DDS QoS deadline 保证信号在设定时间内到达；HTTP + TCP 重传延迟不可控

#### HTTP 不具备的车载能力

- **QoS Deadline**：信号超时未到达自动报警（功能安全相关）
- **自动发现（Dynamic Discovery）**：新节点插上自动发现 Topic，无需配置 IP/Port
- **零拷贝共享内存**：同一 SoC 内跨进程通信，数据不动只传指针
- **空间解耦**：发布者不知道、不关心谁在订阅，系统松耦合，节点可以独立升级和替换

#### 实际分工

- **HTTP**：低频控制指令（开空调、调音量、之前的 `{bool, char[256]}`）——请求-响应模式天然匹配
- **DDS / SOME/IP**：高频信号流（车速、障碍物距离、GPS、激光雷达点云）——发布-订阅 + 实时 QoS 是刚需
- 两者不是替代关系，是共存关系。车里既要地铁（DDS），也要出租车（HTTP）
